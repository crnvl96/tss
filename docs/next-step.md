**Drizzle schema + first migration.** It's the smallest unblocker that the rest of the project is waiting on.

## Why this next

The Docker container being "up" only means you have a Postgres process. The project still has zero tables, no migrations folder, and no schema file. Every downstream piece — the Temporal workflow that writes chunks, the `/search` endpoint that reads them, the `model_version` and `content_hash` columns you documented in `decisions.md` — is gated on a schema existing.

Right now the `db:up` script is the only thing exercising that container, and the only "schema" is the pgvector extension. You can spin the container, but it has nowhere for the chunks to land.

## What "done" looks like for this step

1. `drizzle.config.ts` (pointing at the running container, reading from a new `src/db/schema.ts`)
2. `src/db/schema.ts` with the `chunks` table — `id`, `doc_id`, `chunk_index`, `content`, `embedding vector(384)`, `model_version`, `content_hash` (unique), `created_at`
3. A `migrations/` directory generated by `drizzle-kit generate`
4. `npm run db:migrate` (new script) running `drizzle-kit migrate` against the dev container
5. Verified: `psql` into the container, `\d chunks` shows the table; you can insert a row with a fake vector and read it back
